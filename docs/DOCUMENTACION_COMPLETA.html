1. Qué es el proyecto

El JD Developer CRM es el centro de control de la agencia. En una sola aplicación web permite:

  • Captar leads automáticamente (scraping de Google Maps, web, redes) y desde el formulario de la web pública.
  • Gestionar el pipeline de ventas con un tablero Kanban de arrastrar y soltar.
  • Enviar outreach y responder correos (SMTP/IMAP) con redacción asistida por IA.
  • Organizar campañas, tareas y seguimientos.
  • Analizar métricas (embudo de conversión, forecast por probabilidad).
  • Puntuar oportunidades con IA (Claude asigna un score 0–100).
  • Todo con borrado suave y papelera de recuperación.
Idea clave: no tiene un backend propio programado a mano. El "servidor" es n8n, que hace de proxy seguro contra Google Sheets, la base de datos real del sistema.

3. Arquitectura de conexiones

El corazón del diseño. El dashboard nunca habla directamente con Google Sheets; siempre pasa por webhooks REST de n8n:

┌─────────────────┐ HTTP / Axios ┌──────────────┐ OAuth2 ┌────────────────┐ │ Dashboard CRM │ ───────────────────► │ n8n │ ────────────► │ Google Sheets │ │ (React / Vite) │ webhooks REST │ workflows │ credencial │ (base de datos)│ │ navegador │ ◄─────────────────── │ = "backend" │ ◄──────────── │ pestañas/tabs │ └─────────────────┘ { rows: [...] } └──────┬───────┘ └────────────────┘ │ ┌───────────────────────┼───────────────────────┐ ▼ ▼ ▼ Claude (IA) Apify (scraping) SMTP / IMAP redacción + scoring captación de leads envío/recepción correo

Endpoints principales

  • Lectura: GET /webhook/crm-sheets-read?sheet=:tab
  • Escritura: POST /webhook/crm-sheets-write con cuerpo { action, leadId, ... }

Cada operación de negocio (crear lead, mover etapa, puntuar con IA, enviar correo) es un action distinto que llega al mismo webhook de escritura.

¿Por qué así? Para no incrustar una Google API key en el bundle del frontend. n8n guarda la credencial OAuth2 de Google de forma segura y el navegador solo llama a webhooks. Esto separa las credenciales sensibles del código que se descarga al cliente.

5. Los workflows de n8n (el "backend")

Los flujos viven como JSON en la carpeta n8n/ y se importan a n8n. Cada uno expone webhooks:

WorkflowFunción
CRM_API_Leer_SheetsLee cualquier pestaña como JSON { rows: [...] }
CRM_API_Escribir_SheetsCRUD por action: crear/editar/eliminar leads, pipeline, campañas, contactos, notas, config
CRM_API_Web_LeadGestiona solicitudes del formulario web
CRM_API_TareasCRUD de tareas y seguimientos
Endpoints IA/crm-generate-ai, action: analizar_lead, puntuar_lead → llaman a Claude
Correo/crm-send-reply → envío SMTP con adjuntos (base64)
Captación/crm-buscar-leads → dispara scraping con Apify

El cliente TypeScript que consume todo esto es src/services/crmApi.ts: un objeto con un método por cada acción, todos apoyados en una instancia de Axios con baseURL y un header opcional X-CRM-TOKEN.

6. Cómo fluye una acción (paso a paso)

Ejemplo real: convertir una solicitud web en lead (hook useConvertWebLead):

  1. Un componente React invoca el hook (una mutation de React Query).
  2. El hook llama a crmApi.createLead() + updatePipeline() + updateWebLead() (POST vía Axios a los webhooks).
  3. n8n recibe cada petición y escribe en Google Sheets usando su credencial OAuth2.
  4. Al terminar, React Query invalida las queries ['leads'] y ['web_leads'].
  5. La UI se re-hidrata automáticamente con los datos nuevos.
// Escritura optimista + persistencia
await crmApi.createLead({ leadId, empresa, emailContacto, ... })
await crmApi.updatePipeline({ leadId, estado: 'contactado', ... })
await crmApi.updateWebLead({ id, estado: 'cerrado', etiquetas })
// onSuccess → queryClient.invalidateQueries(['leads'])

9. Gestión de estado y datos

Dos capas complementarias:

  • React Query — estado del servidor: cachea las respuestas de Sheets/n8n, refresca cada 30 segundos (refetchInterval), y tras cada escritura invalida las queries afectadas para mantener la UI sincronizada.
  • Zustand — estado local/global: sesión de usuario, leads hidratados, campañas, estado de UI. La sesión se persiste en localStorage.
Las escrituras son optimistas en el store local: la UI reacciona al instante y, cuando el webhook responde, quedan además persistidas en Sheets.

10. Autenticación

Versión actual (v1): contraseña única validada en el cliente y persistida en localStorage. La estructura ya contempla multiusuario con roles (admin / vendedor / viewer) y 2FA a futuro.

Punto crítico de seguridad: al validarse en el frontend, el login es falsificable. La Fase 2 prevé mover la protección a Cloudflare Access para el equipo interno (gratuito), dejando Supabase para un futuro portal de clientes.

11. Desarrollo vs. Producción

En desarrollo

Vite proxya las rutas /n8n-api y /n8n-hook hacia n8n para evitar CORS y no exponer la API key al navegador:

server.proxy = {
  '/n8n-api':  → n8n /api/v1   // API pública de n8n
  '/n8n-hook': → n8n /webhook  // webhooks del CRM API
}

En producción

  • Se compila estático (tsc -b && vite build) con base: './' para vivir en jddeveloper.com/crm.
  • Se sirve detrás de nginx (ver deploy/), que inyecta X-N8N-API-KEY del lado servidor — así la key sale del bundle.
  • CI/CD con GitHub Actions (.github/workflows/) y contenedor Docker (deploy/Dockerfile).

13. Seguridad y hoja de ruta

Puntos identificados en la auditoría técnica:

  • Crítico Login falsificable (validación en cliente).
  • Crítico API key de n8n en el bundle → se resuelve con proxy nginx server-side.
  • Crítico Webhooks sin autenticación → mitigable con X-CRM-TOKEN.

Fase 2 (planeada): Cloudflare Access para el equipo interno; migración futura de Sheets → Supabase solo cuando se solicite explícitamente; portal de clientes con Supabase Auth.

14. Resumen ejecutivo

Es una SPA React + TypeScript (Vite / Tailwind) que usa React Query + Zustand para el estado y que, en lugar de un backend propio, se comunica por webhooks de n8n. n8n hace de proxy seguro contra Google Sheets (la base de datos) e integra Claude (IA), Apify (scraping) y SMTP/IMAP (correo) — todo para automatizar la captación y la gestión de clientes de la agencia.
